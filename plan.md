# 一番賞虛實整合（O2O）系統設計企劃書

本系統是一款專為實體店家設計的一番賞（Ichiban Kuji）管理與線上抽獎系統。採用前後端分離架構，後端支援 Docker 容器化部署，前端以網頁呈現。核心定位為**「虛實整合的線上電子抽」**，兼顧線上抽獎的便利性與實體店面的營運彈性。

---

## 一、 系統核心亮點與防弊機制

1. **絕對公平與後端洗牌（Shuffle）**
   * **邏輯：** 當管理員成功建立獎池時，後端系統會立刻依據總庫存量，自動生成所有抽獎券的實體陣列（例如：001 到 080 號）並完成隨機洗牌，將其狀態設為「未抽取」並寫入資料庫。
   * **防弊：** 前端使用者僅能看到「剩餘張數」與「剩餘獎項數量」，絕對無法透過 API 逆向查詢未抽取的號碼對應什麼獎等，杜絕店員或駭客內定、偷看大獎的漏洞。
2. **即時庫存同步控制（Concurrency Lock）**
   * 當熱門獎池開放時，系統需處理多人同時點擊同一個號碼或同時抽獎的併發狀況，防止超賣（例如兩個人同時抽中最後一條 A 賞）。

## 二、 網頁管理端功能規範

### 1. 2FA 管理員安全登入
* 管理員登入必須通過雙道身分驗證（2FA，如 Google Authenticator 驗證碼）。
* 用於保障店家的資產安全，尤其是變更店家帳戶、修改獎品內容、重新洗牌、刪除獎池等高風險操作。

### 2. 獎池與獎品管理
* **新增獎池：**
  * **基本資訊：** 獎池名稱、頂部橫幅（可選、可上傳圖片）。
  * **限制條件：** 每個獎池**至少要有兩種等級（例如 A 賞、B 賞）**被新增。
* **獎品等級欄位：**
  * 每一種等級的獎品皆必須填入：【獎項名稱】、【獎項庫存】、【獎項類型（如公仔、吊飾）】、【成本】、【市價】。
* **自動編號與加總：**
  * 系統在獎池與獎品新增成功後，會自動為該獎池及旗下各獎品分配唯一的流水編號（例如：`POOL-XXXX`、`GRADE-XXXX`）。
  * 系統會根據各等級填寫的庫存，自動加總該池的「總抽數」（例如：A賞 2 套 + B賞 8 套 + C賞 70 套 = 總計 80 抽）。
* **商業欄位延伸：**
  * **單抽售價：** 供店家設定這一抽要賣顧客多少錢（例如：單抽 300 元），以便系統自動計算預期營收與利潤率。
  * **最後賞（Last One 賞）配置：** 獨立的勾選區塊，設定當該池最後張牌被抽走時，系統自動配給該位顧客的額外大獎（不計入總抽數中）。

### 3. 物流與兌獎規則設定
店家可在後台針對個別獎池進行物流模式的開關：
* **允許到店自取：** 預設開啟，需設定或連動「門店地址」與「現場核銷期限」。
* **允許寄送：** 店家可自由勾選開啟或關閉。若開啟，可設定基本運費與免運門檻。

### 4. 現場核銷與出貨管理
* **現場核銷介面：** 提供店員掃描顧客出示的動態核銷 QR Code，掃描後系統立刻顯示該顧客中獎的具體項目（例如：A賞x1、E賞x2），店員清點並交付實體獎品後點擊「確認核銷」，狀態即變更為 `已領取`，防止重複領獎。
* **出貨訂單管理：** 當顧客選擇寄送時，後台會獨立生成出貨報表，供店家打包、填寫物流單號（如超商交貨便、郵局單號），並將狀態更新為 `已出貨`。

## 三、 前端使用者端功能規範

### 1. 獎池詳情頁面配置
當使用者點選進入特定獎池後，頁面頂部會呈現**橫幅圖片**，並在其下方清晰呈現以下核心資訊與操作區塊：
* **【單抽定價】：** 清楚標示該獎池單次抽獎的金額（例如：$300 / 抽）。
* **【剩餘抽數】 / 【總抽數】：** 顯示目前進度（例如：剩餘 45 抽 / 總計 80 抽），讓使用者判斷大獎機率。
* **【即時包牌價】：** 系統根據 `剩餘抽數 × 單抽定價` 自動動態計算的總金額。提供一鍵包牌（全開）按鈕，點擊可直接清空該池剩餘所有抽獎券，並確保能獲得最後賞。
* **【預估毛利率】（後台利潤率展示）：** * *註：此欄位實務上通常僅展示於「管理員模式」或「店家端檢視」，若要開放給特定進階使用者，則由系統開關控制。*
* **【各賞配置商品以及數量】（即時殘量看板）：** 條列式或圖卡式呈現 A賞、B賞、C賞等各等級。清楚標示「商品名稱」與「剩餘數量 / 初始數量」（例如：A賞 經典公仔 殘 1 / 共 2 筆）。
* **【開獎紀錄】：** 滾動式跑馬燈或動態列表，即時更新該獎池「幾分鐘前由用戶 XXX 抽中 B賞」的歷史紀錄，增加互動感與信任度。
* **【選號區】：** 棋盤式分格呈現 1 至 N 號（依總抽數而定）的數位抽獎券。
  * **未抽取號碼：** 顯示灰色或可點擊狀態，點擊後會加入購物車準備結帳。
  * **已抽取號碼：** 顯示為已劃掉、變暗或直接顯示該號碼已被抽中的獎項等級（如「05號 已被抽中 A賞」），確保資訊透明。

### 2. 虛擬倉庫（賞物架）機制
* **暫存管理：** 顧客抽中的獎品不會立刻進入物流，而是先放進網頁上的「我的倉庫/賞物架」。顧客可以選擇繼續抽，或累積一定數量後再決定如何領取。
* **兌獎操作選項：**
  1. **申請出貨（寄送）：** 若該獎池有開啟寄送選項，顧客可勾選想要寄送的獎品，合併結帳運費並填寫收件資訊。
  2. **現場兌獎（自取）：** 顧客勾選要到實體店面領取的獎品，系統隨之生成一個**「合併核銷動態 QR Code」**。
* **賞物保留期限：** 系統設定「抽中後 X 天內需兌換/出貨完畢，逾期視同放棄」，避免顧客因抽到小獎故意不領而導致店家倉庫堆積實體貨物。

## 四、 系統核心業務流程 (User Flow)

[店家管理端] 透過 2FA 登入 -> 建立獎池、設定獎品與單抽售價、開啟/關閉寄送 (後端自動洗牌生成 Ticket)
      │
      ▼
[線上前端網頁] 顧客瀏覽獎池詳情 (查看剩餘抽數、即時包牌價、各賞配置殘量) 
      │
      ▼
[選號開獎區] 顧客挑選未抽取的號碼 -> 線上支付/扣點結帳 -> 後端即時鎖定號碼並開獎
      │
      ▼
[後端系統] 扣減庫存、判定中獎獎項 -> 將中獎獎品塞入顧客的 [虛擬倉庫]、更新頁面開獎紀錄
      │
      ├─► 選擇 1：現場兌獎 ──► 產生動態 QR Code ──► 店員掃描核銷確認 ──► 實體發貨完成
      │
      └─► 選擇 2：店家寄送(若有開) ──► 填寫地址付運費 ──► 店家後台出貨並填寫單號 ──► 完成

## 五、 資料庫概念模型（Schema 結構）

- `Pool` (獎池資料表): 記錄獎池編號、名稱、橫幅圖片、單抽售價、是否開放寄送、運費設定、狀態（草稿/上架/完售/下架）。

- `PrizeGrade` (獎品等級資料表): 關聯 Pool。記錄等級名稱（A賞/B賞...）、獎項名稱、初始庫存、目前剩餘庫存、獎品類型、成本、市價。

- `Ticket` (抽獎券資料表): 關聯 Pool 與 PrizeGrade。在獎池建立時，依據總庫存量預先生成 N 筆資料（每一筆都有獨立的流水序號如 1~80），欄位包含 is_drawn（是否已被抽走）、user_id（中獎人 ID）、drawn_at（中獎時間）、serial_number（選號格子對應的號碼）。

- `VirtualWarehouse` (虛擬倉庫資料表): 記錄使用者目前擁有的獎品、來源 Ticket、目前狀態（未領取/申請出貨中/已自取核銷/已寄出/已逾期放棄）。

## 六、 技術棧 (Tech Stack)

* 前端網頁: Vue 3 / Tailwind CSS (託管於 Vercel)
* 後端 API: Python FastAPI (部署於 Render)
* 資料庫: Supabase (PostgreSQL)
* 2FA 驗證: PyOTP 套件 (後端純演算法)
* 圖片儲存: Imgur API 或 Supabase Storage
